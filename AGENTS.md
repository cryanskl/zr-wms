## 你的任务

把一套「汽配仓库管理系统（WMS）」从**已完成的设计与数据库**，实现成**可运行的软件**。
设计阶段已结束：数据库表结构、存储过程、业务逻辑都已写好，并在 PostgreSQL 16 上验证通过；API 契约已定。

## 项目背景（一句话）

给汽配客户做的多仓库存系统，核心诉求是「查得快、查得准」。三个仓库、大量 SKU（原材料 RM / 半成品 SF / 成品 FG / 配件 ACC），目的是把原来在微信群里记账的混乱流程，升级成结构化、可追溯的系统。

## 技术栈（已锁定，不要替换）

- 全栈 **TypeScript**。
- 后端：**Node + NestJS**。
- 查询层：**Drizzle 或 Kysely**（轻量、类型安全）。**不要用 Prisma / TypeORM** —— 逻辑在数据库里，重型 ORM 会和它打架。
- 前端：**React + Vite + Ant Design + TanStack Query**。
- 数据库：**PostgreSQL 15+**（已用到 `NULLS NOT DISTINCT`、生成列等特性，14 及以下不行）。
- 产品图存对象存储（OSS/S3），库里只存 URL。

## 不可违背的架构约束（这些最容易被你「好心」破坏）

1. **数据库是真相层，逻辑在库里。** 应用层不直接拼 SQL 改库存。所有库存写操作（入库/出库/移库/预留/履约/冲正/盘点）必须调对应的 `op_*` 存储过程；所有推衍（缺料/产能/where-used）调 `fn_*` 函数。
2. **绝不在应用层 `UPDATE`/`DELETE` `inventory` 或 `stock_movement`。** 流水只增不删（数据库已用触发器强制），快照只能经存储过程改。
3. **不要重写或「优化」那三个 SQL 文件。** 若你认为有 bug，先告诉我、说明理由，等我确认，**不要擅自改数据模型**。
4. **权限按下面的矩阵，用 NestJS Guard 逐端点落地。** 动作级权限（强制出库、改价格）必须在后端判，不能只靠前端隐藏按钮。
5. **路径别名是派生数据。** 任何改 BOM 的操作之后，要调 `fn_regen_path_aliases()` 重算，别让别名和 BOM 脱节。

## 权限矩阵（来自功能清单）

三角色：操作员 / 管理员 / 老板。

- **三角色都可**：查看库存、新增别名、入库、出库、库存调整、导出 Excel、查看 BOM。
- **管理员及以上**：新增产品、改产品编码（= 产品 ID 维护）、改 BOM、查看价格、自定义仓库/货架结构。
- **仅老板**：改价格。
- **谁都不可**：删除流水（只能作废 / 冲正）。
- **强制出库**（库存不足时）：需管理员。

## 关键业务规则（来自决策记录）

- 内部 ID（RM/SF/FG/配件）**终身不变**；显示别名 / 路径别名与之分离。
- 订单只有两类：**采购**（买原料）、**生产**（客户所需）。生产订单产出成品即结束，**不含出库发货**；采购到货**自动联动入库**。
- **批次只用于原材料**；半成品 / 成品不记批次。
- 库存粒度 = 产品 × 仓库 × 库位 × 质量态（良/不良/不可用）× 批次（仅原料）。**可用 = 在库 − 预留**。
- 外协 = 无货架的内置仓库。
- 低库存预警按**产品总库存**判断。
- 并发十人内：可变表用 `version` 列做乐观锁，冲突提示「刷新后重试」。

## 工作方式（重要）

**一次只做一条「竖切」，从数据库一路打通到界面，不要一次实现所有功能。** 顺序：

1. **铺地基**：建库（按序跑三个 SQL）、插少量测试数据（几个产品 + 别名 + 一棵能体现多级的 BOM）、跑一次 `fn_regen_path_aliases()`。
2. **第一刀 = 搜索**（客户头号诉求）：
   - 后端：`GET /search?q=` → 用 `pg_trgm` 模糊查 产品名 / 别名 / 路径别名 / 备注 四处。
   - 前端：搜索框 + 结果列表（TanStack Query 调）。
   - **先不做鉴权**，先让这条链通。
3. 跑通后，按同样的 `controller → service → 查询` 模式扩展：出入库 → 库存查询 → 订单/预留/缺料推衍 → 盘点/报表。

每完成一刀要**能跑、能演示**，再进下一刀。
